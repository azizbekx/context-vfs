# 🧠 Qontext AI: Enterprise Context-Base System

Welcome to Qontext AI, an advanced, highly-generalized architecture designed to unify disconnected enterprise data silos into an **AI-ready Knowledge Graph and Virtual File System (VFS)**.

This project was built to solve the core problem of AI in the enterprise: LLMs are only as good as the context they are given, but enterprise data is messy, contradictory, and scattered across dozens of platforms (HR systems, ticketing queues, CRMs, email threads).

## 🌟 Key Features

### 1. Declarative Ingestion Schema (`dataset_schema.json`)
Gone are the days of writing custom Python parsers for every new data source. Qontext AI uses a powerful JSON-based schema engine. You simply declare how a raw data file maps to entities, properties, and relationships.
*   **Highly Extensible:** Effortlessly ingest employee records, support tickets, customer chats, and more, just by adding a few lines to the JSON schema.
*   **Provenance Tracking:** Every single fact generated by the ingestion engine traces back to its exact raw source and line number.

### 2. Intelligent Conflict Resolution (Human-in-the-Loop)
Enterprise data is famously contradictory (e.g., HR says "Software Engineer", an internal resume says "SWE", and a project database says "Developer").
*   **LLM Auto-Resolve:** During ingestion, the system pings Gemini (`gemini-3.1-flash-lite`) to determine if conflicting strings are actually functionally identical synonyms. If they are, it auto-merges them!
*   **Human Inspector:** If the conflict is a genuine contradiction (e.g., two different names for the same employee ID), it is queued for Human Review. The React dashboard allows data stewards to click "Resolve" and instantly correct the global truth.

### 3. Real-Time Semantic Vector Search
Keyword matching isn't enough. We've integrated **Gemini Embeddings (`gemini-embedding-2`)** directly into the SQLite database.
*   The system generates vector embeddings for every single entity.
*   Users (and AI agents) can search conceptually (e.g., *"financial data rules"*) and instantly retrieve the exact right entities and markdown files.

### 4. Full AI Interoperability (MCP Server)
The ultimate goal of Qontext is to be the brain for your AI agents.
*   Includes a fully functioning **Model Context Protocol (MCP) Server**.
*   Exposes tools like `search_context`, `get_entity_context`, and `read_vfs_file` directly to your IDE (Cursor) or AI assistants (Claude Desktop).

### 5. Beautiful React Dashboard
A premium UI built to visualize complex relationships easily.
*   **Virtual File System (VFS):** Browse the generated markdown representations of your company data.
*   **Interactive Context Graph:** Click any node to traverse the company knowledge graph and see exact relationship edges (`reports_to`, `manages`, etc.).
*   **Live Search & Inspector:** Type to semantically search, click results to snap the graph to them, inspect provenance, resolve review items, and manually add/edit/delete entities and facts.

## 🚀 Quickstart Guide

### 1. Setup Environment
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
export GEMINI_API_KEY="your-api-key"
```

### 2. Build the Context Graph (Ingestion & Embeddings)
This command reads the raw dataset, applies the built-in extractors plus `dataset_schema.json`, resolves simple conflicts, and generates the VFS. Add `--use-llm` when `GEMINI_API_KEY` is configured and you want Gemini-assisted policy extraction, embeddings, and synonym conflict checks.
```bash
python3 context_base.py build --force
```

### 3. Start the Backend API Server
This hosts the SQLite Knowledge Graph and Search endpoints:
```bash
python3 context_base.py serve
```

### 4. Start the Frontend UI
In a separate terminal, launch the React dashboard:
```bash
cd ui
npm run dev
```
Open `http://localhost:5173` in your browser.

## Demo Script

Run a complete non-interactive demo from a clean output directory:

```bash
bash scripts/demo.sh
```

The script builds the context base, prints graph/review counts, runs a retrieval query, and shows the generated source coverage file. You can also choose a custom output directory:

```bash
bash scripts/demo.sh context_base_demo_out
```

If you already have an indexed output directory and only want to inspect it without rebuilding or regenerating the VFS:

```bash
bash scripts/demo.sh --skip-build context_base_out
```

Then start the product surfaces:

```bash
python3 context_base.py serve --out-dir context_base_demo_out
cd ui
npm run dev
```

## What To Inspect

- `context_base_out/context.db`: SQLite graph with entities, facts, edges, source records, review items, and optional embeddings.
- `context_base_out/vfs/company/index.md`: generated virtual file system entry point.
- `context_base_out/vfs/company/source-coverage.md`: source coverage, stale-source health, entity counts, and facts by source.
- React dashboard: VFS browser, graph inspector, review resolution, and manual CRUD for entities/facts.
- `mcp_server.py`: tools for AI agents: `search_context`, `get_entity_context`, and `read_vfs_file`.

## 🤖 Using the MCP Server

To attach the Qontext brain to your AI agent (like Claude Desktop or Cursor), simply point your MCP configuration to the provided server:

```json
{
  "mcpServers": {
    "qontext-ai": {
      "command": "python3",
      "args": ["/absolute/path/to/mcp_server.py"]
    }
  }
}
```

You can point the MCP server at a non-default build output with either an
environment variable or an argument:

```bash
CONTEXT_BASE_OUT_DIR=context_base_demo_out python3 mcp_server.py
python3 mcp_server.py --out-dir context_base_demo_out
```

Verify MCP setup before connecting an agent:

```bash
python3.10 -m pip install -r requirements.txt
python3.10 mcp_server.py --check-deps --out-dir context_base_out
```

The HTTP API and local tests run on Python 3.9 in this repo, but external MCP
clients require the `mcp` package, which requires Python 3.10 or newer.

Agent tools exposed by the MCP server:

- `get_context_base_status`: graph, source, review, and VFS counts.
- `search_context`: retrieve relevant entities, facts, and VFS files.
- `get_entity_context`: inspect facts and graph neighbors for an entity.
- `read_vfs_file`: read generated markdown from the VFS.
- `list_vfs_files`: list generated VFS files.
- `get_fact_source`: inspect raw source provenance for a fact.
- `list_review_items`: inspect unresolved human-review conflicts.
- `resolve_review_item`: resolve a review item and regenerate the VFS.
- `add_entity_fact`: add a confirmed manual agent/human annotation.

For a repeatable judge workflow, see `SKILLS.md`.
